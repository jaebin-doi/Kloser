# Realtime STT & Voice AI 도입 가이드 2026

> 검증일: 2026-05-07  
> 목적: Kloser의 실시간 상담 보조/통화 분석 제품에 붙일 STT 벤더와 아키텍처를 결정하기 위한 실무 가이드  
> 기준: 공식 문서로 확인 가능한 기능만 확정값으로 기록하고, 성능/가격/한국어 품질은 반드시 POC로 재검증한다.

## 0. 최종 결론

Kloser의 현재 요구사항은 다음에 가깝다.

```text
한국어 상담/회의 음성
→ 낮은 latency
→ partial transcript
→ final transcript 안정화
→ 실시간 AI 제안
→ 이후 통화 이력/검색/CRM 연동
```

따라서 단순 batch transcription API가 아니라 **Realtime Voice Stack** 관점으로 설계해야 한다.

### 추천 의사결정

| 목적 | 1순위 | 2순위 | 3순위 | 비고 |
|---|---|---|---|---|
| 한국어 실시간 STT 안정성 | Azure Speech | NAVER CLOVA Speech | Deepgram Nova-3 | Azure/NAVER는 한국어 안정성 후보, Deepgram은 저지연 후보 |
| 초저지연 voice UX | Deepgram Nova-3 | OpenAI Realtime transcription | Gladia | partial/final/endpointing 실측 필요 |
| AI voice agent | OpenAI Realtime | Azure OpenAI Realtime | Deepgram Flux | STT 단독이 아니라 대화 엔진까지 포함할 때 |
| 국내 공공/금융/한국 리전 | NAVER CLOVA Speech | Azure Speech | - | 데이터 위치/계약 조건 확인 필요 |
| 영어 중심 voice agent | Deepgram Flux / Nova-3 | AssemblyAI Streaming | OpenAI Realtime | AssemblyAI realtime은 한국어 후보에서 제외 |

### Kloser의 1차 POC 권장안

1. **Azure Speech**: 한국어 baseline. `ko-KR`, 실시간 인식, diarization 가능성 확인.
2. **NAVER CLOVA Speech streaming gRPC**: 국내/한국어 특화 baseline. gRPC, PCM 16kHz 제약 확인.
3. **Deepgram Nova-3**: latency baseline. `ko`/`ko-KR`, `interim_results`, endpointing 확인.
4. **OpenAI Realtime transcription**: AI 제안/voice agent 확장 후보. WebRTC/WebSocket, `gpt-4o-transcribe` incremental delta 확인.

Google, Speechmatics, Gladia는 2차 후보로 둔다. AssemblyAI는 공식 streaming 언어 지원 기준으로 한국어 realtime에는 아직 맞지 않는다.

## 1. 시장 판단

과거 STT 시장은 대체로 다음 구조였다.

```text
음성 파일 업로드
→ 서버에서 transcription
→ 최종 텍스트 반환
```

2026년 제품 경쟁은 다음 구조로 이동했다.

```text
마이크/전화 오디오 스트림
→ WebSocket / WebRTC / SDK / gRPC
→ partial transcript
→ endpointing / VAD
→ final transcript
→ LLM 분석/추천
→ 상담 UI 실시간 반영
```

핵심 경쟁 요소도 바뀐다.

| 과거 | 현재 |
|---|---|
| 최종 STT 정확도 | partial 품질 + final 안정화 |
| batch 처리 속도 | first partial latency |
| 파일 업로드 API | WebSocket/WebRTC/gRPC streaming |
| transcript 생성 | turn-taking, interruption, endpointing |
| 단일 STT 모델 | STT + LLM + TTS + UI orchestration |

Kloser는 "실시간 상담 보조" 제품이므로 다음을 별도 지표로 봐야 한다.

- `first_partial_ms`: 첫 중간 결과가 뜨는 시간
- `final_transcript_ms`: 발화가 끝난 뒤 최종 결과가 뜨는 시간
- `endpoint_delay_ms`: 사용자가 말을 멈춘 뒤 turn이 닫히는 시간
- `partial_churn_rate`: 중간 결과가 얼마나 자주 바뀌는지
- `cer_ko`: 한국어 문자 오류율
- `domain_term_hit_rate`: 제품명/회사명/전문용어 인식률
- `reconnect_recovery_ms`: 네트워크 끊김 후 복구 시간

## 2. Whisper 판단

Whisper는 여전히 좋은 batch transcription 모델 계열이지만, Kloser의 primary realtime STT 엔진으로 두면 안 된다.

### 이유

Whisper 기반 pseudo streaming은 보통 이렇게 동작한다.

```text
1~3초 audio chunk
→ Whisper 반복 호출
→ 결과 이어붙이기
```

이 방식은 구현이 쉽지만 다음 한계가 있다.

- 첫 partial latency가 chunk 크기에 묶인다.
- chunk 경계에서 문장이 흔들린다.
- endpointing, interruption, turn-taking을 별도 구현해야 한다.
- 대규모 동시 연결에서 streaming-native 모델보다 운영 난도가 높다.

OpenAI Realtime transcription 문서에서도 `whisper-1`은 delta가 full turn transcript처럼 나오고, `gpt-4o-transcribe` 계열은 incremental transcript를 제공한다고 구분한다. Deepgram 문서도 Whisper 계열은 non-Whisper 모델보다 scale/latency 측면에서 불리하다고 설명한다.

### 결론

Whisper는 다음 용도로만 둔다.

- 녹취 파일 batch 재처리
- 저비용 backfill
- vendor outage 시 fallback
- offline/self-host 연구 후보

실시간 상담 UX의 primary 엔진은 streaming-native STT를 쓴다.

## 3. 후보별 검증 결과

## 3.1 Azure Speech

### 공식 문서로 확인한 사실

- Azure Speech는 real-time transcription, fast transcription, batch transcription의 언어 지원표를 제공한다.
- `ko-KR` Korean은 speech-to-text 지원 목록에 포함된다.
- real-time diarization quickstart가 있고, 발화 구간별 speaker ID를 제공하는 구조를 설명한다.
- Speech SDK 기반 통합이 정석이다.

### 장점

- 한국어 baseline 후보로 가장 보수적이다.
- enterprise 계약, region, SLA, compliance 검토가 쉽다.
- 회의/상담처럼 안정적인 final transcript가 중요한 환경에 맞다.
- diarization, custom speech, language support 문서가 잘 정리되어 있다.

### 단점

- 개발 경험은 WebSocket-first 서비스보다 무겁다.
- browser 직접 연결보다 SDK/서버 orchestration 쪽 설계가 필요하다.
- conversational UX는 OpenAI Realtime처럼 end-to-end voice agent 형태가 아니다.

### Kloser 적용 판단

**1차 POC 필수.** 한국어 상담 품질 baseline으로 둔다.

권장 구조:

```text
browser mic / telephony audio
→ Kloser STT Gateway
→ Azure Speech SDK streaming
→ partial/final transcript
→ Kloser /calls socket
→ live.html transcript + AI suggestion
```

### POC 체크포인트

- `ko-KR` 전화 음질에서 CER
- 회사명/제품명 custom vocabulary 또는 phrase list 효과
- noisy cafe/office 환경
- diarization이 실제 1:1 상담/회의에 충분한지
- endpointing delay 조절 가능성
- 비용 산정: 실제 통화 분당 과금 기준

## 3.2 NAVER CLOVA Speech

### 공식 문서로 확인한 사실

- CLOVA Speech는 Short-form, Long-form, Streaming recognition을 제공한다.
- Streaming recognition은 Korean, English, Japanese를 지원한다.
- 실시간 API는 **gRPC only**로 문서화되어 있다.
- streaming 입력은 현재 PCM headerless raw wave, 16kHz, 1 channel, 16 bits per sample 제약이 있다.
- CLOVA Speech는 Korea region 제공으로 문서화되어 있다.

### 장점

- 한국어/국내 인프라 후보로 강하다.
- 콜센터/국내 고객 데이터 거버넌스 관점에서 검토 가치가 높다.
- batch/long-form까지 같은 벤더로 묶을 수 있다.

### 단점

- WebSocket이 아니라 gRPC라 browser 직접 연결보다 서버 gateway가 필요하다.
- 입력 오디오 포맷 제약이 강하다.
- 해외/다국어 확장성은 Azure/Google/Deepgram보다 좁다.
- 개발자 생태계와 샘플은 글로벌 벤더보다 제한적일 수 있다.

### Kloser 적용 판단

**한국어/국내 리전 요구가 강하면 1차 POC 필수.**

권장 구조:

```text
browser mic
→ Kloser STT Gateway
→ PCM 16kHz mono 변환
→ CLOVA Speech gRPC
→ transcript event normalize
→ /calls socket
```

### POC 체크포인트

- gRPC Node.js client 안정성
- 16kHz mono downsample 품질
- partial/final 이벤트 구조
- endpointing과 finalization 정책
- 한국어 숫자/주소/회사명 인식
- 비용/리전/데이터 보관 계약 조건

## 3.3 Deepgram

### 공식 문서로 확인한 사실

- Deepgram은 live streaming audio 문서를 제공하고, WebSocket/SDK 기반 streaming STT가 핵심이다.
- Nova-3는 general-purpose ASR 후보이며 streaming/batch에 사용 가능하다.
- Nova-3 문서상 `ko`, `ko-KR` Korean 지원이 포함되어 있다.
- `interim_results`, `endpointing`, `speech_final`, `is_final` 같은 realtime UX 제어 옵션이 있다.
- Deepgram은 Flux라는 voice-agent-oriented streaming model도 제공한다. 다만 Flux multilingual 지원 범위와 Korean 지원은 별도 확인이 필요하다.

### 장점

- WebSocket-first라 Kloser의 현재 `/calls` 구조와 잘 맞는다.
- partial/final, endpointing, utterance segmentation을 다루기 좋다.
- latency baseline 후보로 적합하다.
- low-latency 상담 UI 실험 속도가 빠르다.

### 단점

- 한국어 특화 품질은 Azure/NAVER 대비 실측 필요.
- enterprise/regional compliance는 Azure/NAVER보다 별도 검토가 필요하다.
- Flux의 Korean 지원 여부는 현재 문서 기준으로 primary Korean 후보라고 보기 어렵다.

### Kloser 적용 판단

**1차 POC 필수.** Azure/NAVER 대비 latency와 partial UX를 비교한다.

권장 구조:

```text
browser mic
→ Kloser STT Gateway
→ Deepgram live WebSocket
→ interim/final transcript normalize
→ /calls socket
```

### POC 체크포인트

- `model=nova-3`, `language=ko-KR`
- `interim_results=true`
- `endpointing=300/500/700` 비교
- `speech_final`과 `is_final` 의미 분리
- 한국어 domain term boosting 효과
- 긴 상담 연결에서 keepalive/reconnect 안정성

## 3.4 OpenAI Realtime / OpenAI transcription

### 공식 문서로 확인한 사실

- OpenAI Realtime transcription은 transcription session을 만들고 WebSocket 또는 WebRTC로 연결한다.
- `gpt-4o-transcribe`와 `gpt-4o-mini-transcribe`는 incremental transcript delta를 제공한다.
- `whisper-1`은 delta도 full turn transcript처럼 다뤄진다.
- browser/mobile 클라이언트는 WebRTC와 ephemeral key 흐름이 권장된다.
- Realtime API는 STT 단독보다 multimodal/voice conversation 구조에 가깝다.

### 장점

- AI 제안, 음성 agent, interruption, conversation UX 확장에 가장 자연스럽다.
- transcript 이후 LLM 분석으로 넘어가는 비용이 낮다.
- future roadmap상 Kloser가 "AI 상담 코파일럿"에서 "AI 상담원"으로 가면 핵심 후보가 된다.

### 단점

- pure STT만 보면 과한 구조일 수 있다.
- 한국어 고유명사/숫자/현장 소음은 Azure/NAVER와 비교 실측이 필요하다.
- WebRTC/ephemeral token 설계가 들어가면 현재 `/calls` socket 구조와 별도 signaling 계층이 필요할 수 있다.
- 가격은 token/audio model 단위라 통화 분당 원가 산정이 더 복잡하다.

### Kloser 적용 판단

**STT baseline이 아니라 AI voice UX 확장 후보로 POC.**

현재 Phase 1~4에서는 Deepgram/Azure/NAVER처럼 transcript event를 우리 socket으로 normalize하는 구조가 더 단순하다. OpenAI Realtime은 Phase 5+에서 "AI가 직접 듣고 말하는" 방향을 검토할 때 별도 branch로 실험한다.

권장 구조 A: pure transcription

```text
browser mic
→ Kloser STT Gateway
→ OpenAI Realtime transcription WebSocket
→ transcript delta/completed normalize
→ /calls socket
```

권장 구조 B: voice agent

```text
browser WebRTC
→ Kloser ephemeral token endpoint
→ OpenAI Realtime WebRTC
→ data channel events
→ Kloser server event mirror / audit
```

### POC 체크포인트

- `gpt-4o-transcribe` vs `gpt-4o-mini-transcribe`
- Korean partial churn
- final transcript 안정화
- domain prompt 효과
- WebRTC direct 연결 시 감사 로그/저장 구조
- 분당 원가 산정 방식

## 3.5 Google Cloud Speech-to-Text

### 공식 문서로 확인한 사실

- Cloud Speech-to-Text V2에는 Chirp 2 모델이 있고, streaming recognition API가 있다.
- Chirp 2는 word-level timestamps, model adaptation, speech translation 등 기능을 제공한다.
- 문서상 streaming support는 language support가 제한적이라고 표시된다.

### 장점

- 글로벌/다국어 인프라가 강하다.
- Google Cloud ecosystem을 이미 쓰는 팀이면 운영 편의가 있다.
- Chirp 계열은 범용 ASR 후보로 검토 가치가 있다.

### 단점

- Kloser의 1차 목적이 한국어 실시간 상담이라면 Azure/NAVER/Deepgram보다 우선순위가 낮다.
- Chirp 2 streaming에서 Korean/region/model 조합은 POC 전 공식 문서와 콘솔에서 재확인이 필요하다.

### Kloser 적용 판단

**2차 후보.** 글로벌 다국어 요구가 강해지면 POC에 포함한다.

## 3.6 Speechmatics

### 공식 문서로 확인한 사실

- Speechmatics는 realtime transcription, batch transcription, voice agent SDK, TTS, on-prem deployment를 제공한다고 문서화되어 있다.
- managed SaaS와 on-prem 옵션이 있다.

### 장점

- on-prem/private deployment 요구가 생기면 강한 후보.
- 방송/회의/다국어 transcription 계열에서 검토 가치가 있다.

### 단점

- Kloser의 1차 한국어 상담 POC에서는 Azure/NAVER/Deepgram보다 우선순위가 낮다.
- 한국어 realtime 품질과 비용은 별도 실측 필요.

### Kloser 적용 판단

**2차 후보.** 보안상 on-prem 필요가 생기면 다시 본다.

## 3.7 Gladia

### 공식 문서로 확인한 사실

- Gladia는 live transcription API를 제공한다.
- session init API가 WebSocket URL과 session ID를 반환한다.
- partial transcripts를 설정으로 받을 수 있다.
- 단일 realtime session은 3시간 제한이 있다.
- reconnect continuity를 고려한 session URL 구조를 문서화한다.

### 장점

- WebSocket 기반 통합이 명확하다.
- partial/final, reconnect, multi-channel 같은 실무 기능이 문서화되어 있다.
- 다국어/회의형 제품 후보로 검토 가치가 있다.

### 단점

- 한국어 상담 품질은 별도 검증 필요.
- 국내 enterprise/data residency 요구에는 Azure/NAVER보다 불리할 수 있다.

### Kloser 적용 판단

**2차 후보.** Deepgram과 같은 WebSocket UX 후보군으로 비교한다.

## 3.8 AssemblyAI

### 공식 문서로 확인한 사실

- AssemblyAI Streaming STT는 WebSocket 기반 live transcription을 제공한다.
- 공식 문서상 streaming STT는 English 중심이며, multilingual streaming은 English, Spanish, French, German, Italian, Portuguese 범위로 안내된다.
- `language_code`는 async STT와 streaming STT에서 다르게 취급된다.

### Kloser 적용 판단

**한국어 realtime STT 후보에서 제외.**

영어 voice agent나 영어 회의록 제품이면 별도 검토할 수 있지만, Kloser의 한국어 상담 요구에는 맞지 않는다.

## 4. Kloser 권장 아키텍처

## 4.1 원칙

브라우저가 STT 벤더 API key를 직접 들고 있으면 안 된다. 벤더가 browser direct를 지원하더라도 다음 이유로 Kloser 서버가 session/token 발급과 event mirror를 관리한다.

- API key 보호
- 고객/조직별 과금 추적
- transcript 저장/삭제 정책
- audit log
- vendor swap 가능성
- RLS org context 연결
- 장애 시 fallback

## 4.2 표준 구조

```text
platform/live.html
  └─ mic capture / audio worklet
      └─ /calls Socket.io auth session
          └─ server/src/stt/*
              ├─ Azure adapter
              ├─ Naver Clova adapter
              ├─ Deepgram adapter
              └─ OpenAI adapter
                  └─ normalized events
                      ├─ transcript.partial
                      ├─ transcript.final
                      ├─ speech.started
                      ├─ speech.ended
                      ├─ stt.error
                      └─ stt.metrics
```

## 4.3 Normalized event contract

벤더별 payload를 UI가 직접 알면 안 된다. Kloser 내부 이벤트는 고정한다.

```ts
type SttPartial = {
  type: "transcript.partial";
  callId: string;
  seq: number;
  text: string;
  speaker?: string;
  startedAtMs?: number;
  endedAtMs?: number;
  vendor: "azure" | "naver" | "deepgram" | "openai" | "google" | "gladia";
  vendorMeta?: unknown;
  receivedAt: number;
};

type SttFinal = {
  type: "transcript.final";
  callId: string;
  seq: number;
  text: string;
  speaker?: string;
  confidence?: number;
  startedAtMs?: number;
  endedAtMs?: number;
  vendor: string;
  vendorMeta?: unknown;
  receivedAt: number;
};
```

## 4.4 Audio format policy

Kloser 내부 표준 입력은 우선 다음으로 둔다.

```text
PCM16
16kHz
mono
20~100ms frame
```

이유:

- NAVER CLOVA streaming이 16kHz mono PCM 제약을 가진다.
- AssemblyAI/Gladia/Deepgram도 PCM 계열 처리가 쉽다.
- telephony와 bridge하기 쉽다.

다만 browser에서 Opus/WebRTC 경로를 쓸 때는 별도 adapter에서 decode/resample한다.

## 5. POC 벤치마크 설계

## 5.1 테스트 세트

최소 60개 샘플을 만든다.

| 세트 | 개수 | 내용 |
|---|---:|---|
| clean office | 10 | 조용한 사무실 한국어 상담 |
| noisy office | 10 | 주변 대화/키보드/공조음 |
| phone-like | 10 | 8kHz/16kHz 전화 음질 |
| numbers | 10 | 전화번호, 금액, 날짜, 주소 |
| domain terms | 10 | Kloser, CRM, SaaS, 제품명, 회사명 |
| interruption | 10 | 말 끊기, 겹침, 짧은 대답 |

## 5.2 측정 지표

| 지표 | 설명 | 목표 |
|---|---|---|
| `first_partial_ms` | 발화 시작 후 첫 partial까지 | P50 < 500ms |
| `final_after_silence_ms` | 침묵 후 final까지 | P50 < 900ms |
| `cer_ko` | 한국어 문자 오류율 | 후보 간 상대 비교 |
| `number_accuracy` | 숫자/날짜/금액 정확도 | 95%+ 목표 |
| `domain_term_accuracy` | 전문용어 인식률 | boost 전후 비교 |
| `partial_churn_rate` | partial 변경 빈도 | 낮을수록 UI 안정 |
| `disconnect_recovery_ms` | 재연결 후 복구 | 2초 이내 목표 |
| `cost_per_hour_krw` | 실제 과금 환산 | 월 사용량 기준 |

## 5.3 합격 기준

1차 도입 후보는 다음을 만족해야 한다.

- 한국어 상담 30분 연속 streaming에서 끊김 없음
- P50 first partial < 500ms
- P95 final after silence < 1500ms
- 숫자/주소/회사명 오류가 운영상 교정 가능한 수준
- reconnect/finalize 로직이 명확함
- 서버 gateway에서 org/call/user 단위 비용 로깅 가능

## 6.1 Phase 1 Step 4 이후 바로 할 일

현재 Step 4는 인증된 `/calls` WebSocket과 live UI를 만든다. 실제 STT는 다음 phase에서 아래 순서로 붙인다.

3. `server/src/stt/adapters/deepgram.ts`

4. `server/src/stt/adapters/azure.ts`
   - Speech SDK baseline
5. `server/src/stt/adapters/naverClova.ts`
   - gRPC baseline
6. `server/src/ws/calls.ts`
   - `start_call` 시 adapter session 시작
   - `text_chunk` fixture echo 제거
   - audio chunk 입력 이벤트 추가

## 6.2 Adapter interface 초안

```ts
export interface SttAdapter {
  readonly vendor: string;

  start(input: {
    callId: string;
    orgId: string;
    language: "ko-KR";
    sampleRate: 16000;
    channels: 1;
    onEvent: (event: SttEvent) => void;
    onError: (error: SttError) => void;
  }): Promise<SttSession>;
}

export interface SttSession {
  sendAudio(chunk: Buffer): void | Promise<void>;
  finalize(): Promise<void>;
  close(): Promise<void>;
}
```

## 7. 가격 판단 방법

가격은 문서보다 더 자주 바뀐다. 이 문서에는 숫자를 고정하지 않는다.

도입 직전 반드시 다음을 확인한다.

- 분당/시간당 과금인지
- connection open 시간 기준인지 audio duration 기준인지
- partial transcript/diarization/custom vocabulary가 추가 과금인지
- silence 구간도 과금되는지
- multi-channel 과금 방식
- batch와 realtime 가격 차이
- 무료 tier 또는 commit discount
- 한국 리전/enterprise 계약 단가

월 비용 계산식:

```text
월 STT 비용 =
  상담원 수
  × 일평균 통화 시간(분)
  × 월 근무일
  × realtime 분당 단가
  × safety factor(1.2~1.5)
```

## 8. 보안/컴플라이언스 체크리스트

STT 벤더 선정 전 다음 질문을 반드시 확인한다.

- 오디오 원본이 저장되는가?
- transcript가 모델 학습에 사용되는가?
- retention 기간은 얼마인가?
- region 선택이 가능한가?
- 한국 리전 또는 국내 보관이 가능한가?
- DPA/BAA/enterprise 계약이 가능한가?
- subprocessor 목록이 공개되는가?
- API key rotation과 audit log가 가능한가?
- PII redaction 기능이 있는가?
- 고객 요청 시 삭제 API가 있는가?

Kloser 내부 정책:

- browser는 STT vendor API key를 절대 보유하지 않는다.
- 모든 STT session은 `org_id`, `call_id`, `user_id`에 묶는다.
- transcript 저장은 RLS org context를 통과해야 한다.
- raw audio 저장은 기본 off. 저장 필요 시 별도 opt-in과 retention 정책을 둔다.

## 9. 현재 추천 순위

### 한국어 실시간 상담 STT

1. **Azure Speech**
   - 가장 안정적인 baseline 후보.
   - enterprise/diarization/언어 지원 검토가 쉽다.

2. **NAVER CLOVA Speech streaming**
   - 국내/한국어/리전 요구가 강할 때 핵심 후보.
   - gRPC/PCM 제약을 gateway에서 흡수해야 한다.

3. **Deepgram Nova-3**
   - latency와 realtime UX 비교 후보.
   - 한국어 품질은 실측으로 판단한다.

4. **OpenAI Realtime transcription**
   - STT 단독보다 AI voice agent 확장 후보.

### 제외/후순위

- **AssemblyAI Streaming**: 한국어 realtime 지원 범위가 맞지 않아 제외.
- **Google Chirp 2**: 글로벌 후보지만 Korean streaming 조합 재확인 후 2차 POC.
- **Speechmatics**: on-prem/private deployment 요구 시 재검토.
- **Gladia**: WebSocket UX 후보지만 한국어 품질 실측 전까지 2차.
