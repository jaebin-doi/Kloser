# Kloser Phase 2 사용자 가이드

> Phase 2는 Phase 1의 계정·조직·권한 기반 위에 **첫 번째 실제 비즈니스 데이터(고객)**를 올린 단계입니다. 평가자가 직접 고객을 입력·수정·삭제·검색할 수 있게 됐고, 그 데이터는 조직별로 격리된 채 영속 저장됩니다.

---

## 1. Phase 2의 목적

Phase 1에서 안전하게 쓸 수 있는 기반(로그인·조직 분리·권한·세션·인증된 실시간 연결)이 갖춰졌지만, 그 위에 올라간 **실제 업무 데이터는 아직 하나도 없었습니다.** Phase 2는 그 첫 데이터인 **고객 (customers)**을 다음 다섯 가지 약속과 함께 올립니다.

1. **평가자가 화면에서 직접 입력한 고객은 실제로 저장된다.**
2. **저장된 고객은 다음에 다시 들어와도 그대로 보인다.**
3. **다른 조직의 고객은 어떤 방식으로도 보이지 않는다.**
4. **권한에 따라 작성·수정·삭제 가능 여부가 정확히 갈린다.**
5. **잘못된 입력은 화면에서 받아주지 않고, 받아주더라도 서버가 한 번 더 거른다.**

Phase 2가 끝나면, 화면 위 "예시 데이터"는 사라지고 평가자가 만든 데이터로 대체됩니다.

---

## 2. Phase 2에서 가능해진 것

평가 또는 검토 시점에 다음을 직접 확인할 수 있습니다.

- **고객 목록 보기** — Acme 12명, Beta 12명 (조직별 다른 사람) 시드.
- **고객 추가** — 모달에서 이름·회사·이메일·전화·상태·담당자를 입력해 저장.
- **고객 수정** — 같은 모달이 기존 값을 미리 채운 채 열리고, 변경한 값만 반영.
- **고객 삭제** — 확인 후 목록에서 사라짐. 데이터는 시스템 내부에서 보존(soft delete).
- **검색 · 필터 · 정렬** — 이름·회사·이메일 검색 + 상태별 필터 + 정렬. 선택한 조건은 URL에 그대로 반영되어 새로고침·뒤로가기·북마크에서도 그대로 복원.
- **4가지 지표 카드 (KPI)** — 전체 / 활성 / 검토중 / 대기. 추가·수정·삭제 직후 자동 갱신.
- **사이드바 카운트** — 좌측 네비게이션의 고객 수 배지가 실제 데이터와 항상 일치.
- **다른 조직 사용자로 다시 로그인** — 같은 페이지에 들어가도 자기 조직 고객만 보임.

---

## 3. 고객 데이터의 구조

한 명의 고객은 다음 정보를 가질 수 있습니다.

| 항목 | 설명 | 비고 |
|---|---|---|
| 이름 | 고객 담당자 이름 | 필수 |
| 회사 | 소속 회사명 | 선택 |
| 이메일 | 연락 이메일 | 선택, 형식 검증 |
| 전화 | 연락 전화번호 | 선택 |
| 상태 | `활성` · `검토중` · `대기` 중 하나 | 기본값 `대기` |
| 담당자 | 우리 회사 안에서 이 고객을 맡은 사용자 | 선택, 사용자 떠나면 자동 해제 |
| 마지막 연락 | 마지막 통화·미팅 시각 | 선택, Phase 4에서 통화 기록과 자동 연동 예정 |

이 모델은 다음 두 가지 도메인 분리를 따릅니다.

- **고객의 "상태"는 영업 진행 단계가 아니라 행정 분류**입니다 (`활성`/`검토중`/`대기`). 영업 단계(리드 → 자격 검증 → 제안 → 성사/실패)는 Phase 3+에서 별도로 도입됩니다.
- **고객 row는 Kloser 구독 플랜 정보를 가지지 않습니다.** Kloser 자체의 구독 플랜(Starter / Pro / Enterprise)은 조직 단위 정보이고, 고객은 사용사가 관리하는 영업 대상이라는 별개 도메인입니다. Phase 2 진행 중에 발견된 단어 충돌(고객에게 plan 컬럼이 있어 헷갈림)을 해소하기 위해 해당 컬럼을 영구 제거했습니다.

---

## 4. 조직별 데이터 분리 (재확인)

Phase 1이 만든 조직 분리 약속은 고객 데이터에도 그대로 적용됩니다.

- **다른 조직의 고객은 어떤 방식으로도 접근되지 않습니다.** URL 직접 접근, ID 추측 접근 모두 거부됩니다.
- **분리는 두 곳에서 검증됩니다.** 서버에서 사용자의 조직을 확인하고, 데이터베이스가 자체적으로 한 번 더 걸러냅니다.
- **자기 조직에 새 고객을 만들 때도 다른 조직 ID를 박아 넣을 수 없습니다.** 데이터베이스가 시스템 차원에서 거부합니다.

평가 시점에 Acme 계정으로 로그인하면 Acme의 12명만, Beta 계정으로 로그인하면 Beta의 12명만 보입니다. 한 화면에 두 조직 데이터가 섞여 보이는 일은 없습니다.

---

## 5. 권한별 가능한 작업

Phase 1에서 정의한 4단계 권한이 고객 작업에 다음과 같이 적용됩니다.

| 권한 | 목록 보기 | 추가 / 수정 / 삭제 |
|---|---|---|
| 관리자 (Admin) | ✓ | ✓ |
| 매니저 (Manager) | ✓ | ✓ |
| 직원 (Employee) | ✓ | ✓ |
| 조회 (Viewer) | ✓ | **거부** (서버에서 403) |

**조회(Viewer) 권한은 추가·수정·삭제 시도가 화면이 아닌 서버에서 막힙니다.** 화면 비활성화에만 의존하지 않으므로, 직접 요청을 보내도 통과하지 않습니다.

Phase 2 시점에는 매니저 / 직원 모두 조직 전체 고객을 작성·수정할 수 있도록 단순화했습니다. "매니저는 자기 팀 고객만, 직원은 자기 담당 고객만"으로 좁히는 정책은 Phase 3 (팀 초대 활성화 시점)에서 도입됩니다.

## 6. 화면 동작 — 검색·필터·정렬·페이지

### 검색

상단 검색창에 이름·회사·이메일의 일부를 입력하면 즉시 결과가 좁혀집니다. 대소문자 구분 없이 부분 일치합니다.

### 상태 필터

`전체 / 활성 / 검토중 / 대기` 칩으로 한 가지 상태만 보거나 전체를 봅니다.

### 정렬

`이름 / 마지막 연락 / 추가 시각`을 기준으로 정렬합니다. 기본값은 추가 시각의 내림차순 — 가장 최근에 추가된 고객이 가장 위에 보입니다.

### URL 동기화

선택한 검색어·필터·정렬은 URL에 그대로 반영됩니다.

```
/platform/customers.html?q=김&status=active&sort=last_contacted_at&dir=desc
```

이 URL을 북마크하거나 다른 사람에게 보내도 같은 화면이 그대로 열립니다. 새로고침·뒤로 가기에도 화면 상태가 보존됩니다.

### 4가지 지표 카드 (KPI)

`전체 / 활성 / 검토중 / 대기` 카드는 **현재 보이는 목록이 아닌 조직 전체 고객 기준**으로 갱신됩니다. 따라서 필터로 활성만 보고 있더라도 "전체 13명 / 대기 3명" 같은 숫자가 정확히 표시됩니다 — 필터는 보기에만 영향을 주고, 통계는 조직 전체의 진실값입니다.

### 추가·수정·삭제 후 동작

추가·수정·삭제가 성공하면 화면은 **목록 전체를 다시 받아옵니다.** 부분 갱신이 아닌 전체 재조회를 택한 이유는 검색·필터·정렬이 활성화된 상태에서도 KPI·정렬·페이지가 어긋나지 않게 하기 위함입니다. 추가 비용은 사용자에게 보이지 않을 만큼 짧습니다.

## 7. 입력 검증과 안전한 기본값

### 화면에서의 1차 검증

이름이 비어 있으면 저장 버튼이 비활성화됩니다. 이메일은 형식이 어긋나면 입력 직후 표시가 바뀝니다.

### 서버에서의 2차 검증 (실제 안전선)

화면 검증을 우회해서 직접 잘못된 요청을 보내도 서버가 다시 거릅니다.

- 이름이 비어 있는 추가 요청 → `400 invalid_input` (`name`).
- 형식이 맞지 않는 이메일 → `400 invalid_input` (`email`).
- 알 수 없는 상태 값 (`active`/`review`/`pending` 외) → `400 invalid_status`.
- 잘못된 정렬 컬럼 / 비정상 페이지 크기 (음수, 너무 큼) → `400 invalid_<항목>`.
- 다른 조직 ID로 새 고객을 만들려는 시도 → 데이터베이스가 거부.
- 다른 조직의 고객 ID로 수정·삭제 → "찾을 수 없음" 응답 (존재 자체를 노출하지 않음).

서버는 검증 실패를 항목 단위로 명시해 알려줍니다. 화면은 그 정보를 그대로 사용자에게 표시할 수 있습니다.

### 보호되는 형식 동기화

화면 코드와 서버 코드가 같은 입력 형식을 약속하도록 **양쪽 정의를 자동으로 맞춰주는 검증 테스트**가 추가됐습니다. 어느 한쪽만 바뀌면 그 테스트가 실패해서 회귀를 차단합니다.

---

## 8. 데이터 보존 — 삭제는 즉시 사라지지 않습니다

고객을 삭제하면 화면에서는 즉시 사라지지만, 시스템 내부에서는 **삭제 표시만 붙은 채 보관**됩니다(soft delete). 평소 화면에는 절대 노출되지 않지만 운영 차원의 복구·감사가 가능합니다.

- 일반 화면 / 검색 / 통계 어디에도 노출되지 않습니다.
- 다른 사용자가 같은 고객을 다시 만들어도 충돌하지 않습니다 (별개 row로 시작).
- Phase 4+ 운영 단계에서 "복원" UI가 도입되기 전까지는 운영자만 시스템 차원에서 복구할 수 있습니다.

---

## 9. 자동 회귀 — 다음 변경이 이전 약속을 깨지 않도록

Phase 2의 위 약속들은 **자동 회귀 검사**로 매번 검증됩니다.

- **단위 테스트** 65개 — 인증·조직 격리·고객 저장소·고객 API·실시간 연결 인증 합산.
- **고객 API 자체 테스트** 16개 — 정상 흐름과 모든 4xx 에러 경로.
- **고객 흐름 e2e 7 시나리오** — 로그인 → 목록 → 추가 → 수정 → 삭제 → 다른 조직 격리 → 필터 정합. 마지막 시나리오는 앞선 시나리오들이 만든 임시 데이터가 모두 정리됐는지 직접 검사합니다.
- **Phase 1 통화 흐름 e2e 16건** — Phase 2 변경이 통화 흐름을 깨뜨리지 않았는지 회귀 보호.
- **양쪽 형식 동기화 검사** — 서버와 화면의 고객 입력 형식이 같은 정의를 따르는지 확인.

이 다섯 가지가 모두 통과해야만 Phase 2가 합격으로 인정됩니다.

## 10. 사용자가 알아야 할 제한사항

Phase 2는 첫 번째 실제 데이터를 올린 단계입니다. 다음은 아직 다음 Phase로 미뤄져 있습니다.

- **행 클릭 상세 패널** — 고객 한 명의 통화 이력·메모·To-Do는 Phase 4에서 통화 기록과 함께 결합됩니다.
- **자가 회원가입 / 비밀번호 재설정 / 직원 초대 메일** — 현재는 사전 발급 계정으로만 로그인. Phase 3에서 활성화.
- **팀 단위 권한 분기** — "매니저는 자기 팀만, 직원은 자기 담당만" 정책은 Phase 3 (팀 초대 활성화)에서 도입.
- **CSV 일괄 가져오기 / 외부 시스템 동기화 (HubSpot, Salesforce 등)** — Phase 6+ 운영 단계.
- **고급 검색 (오타 허용·동의어·전문 검색)** — 현재는 부분 일치 검색. 데이터가 늘면 Phase 5+에서 인덱스 보강.
- **변경 이력 / 감사 로그** — 누가 언제 무엇을 바꿨는지 추적은 Phase 4 (통화 기록과 같이) 또는 별도 운영 단계.
- **삭제된 고객 복원 UI** — Phase 4+ 관리자 콘솔.
- **결제 / 구독 / 사용량 추적** — Phase 6+ 운영 단계.

## 12. 기술 요약 (참고)

이 섹션은 보안·아키텍처에 관심 있는 평가자가 핵심 결정 사항을 한눈에 보기 위한 짧은 요약입니다. 본 문서의 다른 섹션은 이 용어를 몰라도 이해할 수 있도록 작성했습니다.

- **고객 저장 구조** — `customers` 테이블 + 조직 단위 행 격리(RLS) + 5개 부분 인덱스(`WHERE deleted_at IS NULL`). soft delete는 `deleted_at` 컬럼으로 표현.
- **조직 격리** — Phase 1과 같은 `current_app_org_id()` 기반 4개 RLS 정책 (SELECT / INSERT / UPDATE / DELETE). 새 row 작성 시점에도 `WITH CHECK`으로 다른 조직 ID 작성 차단.
- **권한 분기** — `requireAuth` + `requireRole` 미들웨어 조합. viewer는 쓰기 endpoint에서 application 단계 403, 추가로 RLS가 두 번째 안전선.
- **입력 검증** — `zod` 스키마가 서버 측 단일 진입점. `ZodError → 400 invalid_input + flatten()`, 정렬·페이지 같은 list option 위반은 `400 invalid_<field> + value`로 별도 매핑.
- **양쪽 형식 동기화** — `server/src/types/customers.ts` (zod 원본) + `platform/types/customers.js` (JSDoc 사본) + `test/sync_shared_types.mjs` (정규식 diff 검사). 한쪽만 바뀌면 검사 실패.
- **삭제 정책** — soft delete (`deleted_at` 표시). 모든 인덱스가 `WHERE deleted_at IS NULL` 부분 인덱스라 삭제된 row는 검색·통계에서 자연스럽게 제외.
