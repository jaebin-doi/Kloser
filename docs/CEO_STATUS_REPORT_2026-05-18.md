# Kloser 현재 상황 보고서

보고일: 2026-05-18
보고 대상: 대표님

---

## 1. 결론부터 말씀드리면

Kloser는 **SaaS 기본 뼈대는 많이 만들어진 상태**입니다.

지금까지 만들어진 것은 로그인, 고객 관리, 팀 관리, 통화 기록, 보고서, 보안, 감사 로그, 요금제 제한 같은 기본 운영 기능입니다.

하지만 중요한 점이 있습니다.

**실제 AI 기능은 아직 완성된 상태가 아닙니다.**

현재 AI는 "붙일 준비"가 된 수준입니다. 실제 고객 통화에서 음성을 인식하고, 내용을 요약하고, 상담원에게 답변을 추천하는 기능은 아직 운영 검증이 끝나지 않았습니다.

또 하나 중요한 운영 원칙이 있습니다.

**우리 인프라는 public cloud를 쓰지 않고 온프레미스로 갑니다.**

즉, 서버, DB, Redis, 파일 저장소, 녹취 저장소는 회사가 관리하는 온프레미스 환경 기준으로 잡아야 합니다.

## 2. 현재 상태를 쉽게 정리하면

- 기본 서비스 틀: 많이 만들어짐
- 계정 / 권한 / 보안: 많이 만들어짐
- 고객 / 팀 / 통화 기록: 많이 만들어짐
- 보고서: 기본 기능 있음
- 감사 로그: 있음
- 요금제 제한: 있음
- 실제 AI 기능: 아직 남음
- 실제 결제 연동: 아직 남음
- 통화 녹음 기능: 이제 시작함
- 인프라 방향: 온프레미스

## 3. 지금까지 완료된 것

### 3.1 계정과 보안

다음 기능은 대부분 구현되어 있습니다.

- 회원가입
- 로그인
- 로그아웃
- 이메일 인증
- 비밀번호 재설정
- 직원 초대
- 관리자 / 매니저 / 직원 권한 구분
- 2단계 인증(MFA)
- 회사별 데이터 분리

쉽게 말하면, **회사별로 계정을 만들고, 직원을 초대하고, 권한을 나눠서 쓰는 기본 구조는 갖춰졌습니다.**

### 3.2 고객 관리

다음 기능이 준비되어 있습니다.

- 고객 등록
- 고객 목록 조회
- 고객 정보 수정
- 고객 삭제
- 고객 상태 관리

즉, **CRM의 기본 고객 관리 기능은 들어가 있습니다.**

### 3.3 팀 관리

다음 기능이 준비되어 있습니다.

- 팀 생성
- 팀 수정
- 팀 삭제
- 직원 역할 변경
- 직원 비활성화
- 마지막 관리자 보호

즉, **회사 내부 사용자와 팀을 관리하는 기능은 어느 정도 완성되어 있습니다.**

### 3.4 통화 기록

다음 기능이 준비되어 있습니다.

- 통화 생성
- 통화 종료
- 통화 메모 작성
- 통화와 고객 연결
- 통화 transcript 저장
- 통화 후 할 일 관리

즉, **통화 내용을 기록하고, 나중에 조회할 수 있는 기본 구조는 있습니다.**

### 3.5 보고서

다음 기능이 준비되어 있습니다.

- 팀 보고서
- 기간별 필터
- 상담원별 분석
- 최근 통화 목록
- dashboard 요약

즉, **관리자가 팀 성과를 볼 수 있는 기본 보고서 기능은 있습니다.**

### 3.6 이메일 발송 구조

다음 이메일을 실제 발송 시스템에 연결할 수 있는 구조가 만들어졌습니다.

- 회원가입 인증 메일
- 비밀번호 재설정 메일
- 직원 초대 메일

단, 실제 운영 도메인으로 발송 테스트는 아직 남아 있습니다.

### 3.7 감사 로그

관리자가 나중에 확인할 수 있도록 주요 행동이 기록됩니다.

예:

- 로그인
- MFA 변경
- 직원 초대
- 권한 변경
- 고객 생성/수정/삭제
- 통화 생성/종료
- 보고서 조회
- 요금제 관련 정보 수정

즉, **누가 언제 중요한 행동을 했는지 추적할 수 있는 기반이 있습니다.**

### 3.8 요금제 제한

Starter / Pro / Enterprise 같은 요금제별 제한 구조가 들어갔습니다.

예:

- 직원 수 제한
- 고객 수 제한
- 지식 베이스 수 제한
- 월 통화 수 제한

단, 실제 카드 결제나 구독 결제는 아직 연결되지 않았습니다.

## 4. AI 기능의 정확한 상태

현재 AI는 **완성된 기능이 아닙니다.**

지금 있는 것은:

- AI를 붙일 수 있는 코드 구조
- 테스트용 가짜 AI 응답
- AI 사용량을 기록하는 구조
- AI 비용을 계산할 수 있는 구조
- STT 비용 검토 문서: `docs/research/AZURE_SPEECH_COST_GUIDE_2026.md`

STT 방향:

- STT는 기존 Clova 기준이 아니라 **Azure Speech 기준으로 검토**합니다.
- 비용 기준 문서는 `AZURE_SPEECH_COST_GUIDE_2026.md`입니다.
- 다만 Azure Speech는 기본적으로 외부 클라우드 서비스입니다.
- 우리 인프라 원칙이 "전부 온프레미스"라면, Azure Speech를 실제 운영에 쓰는 것은 원칙과 충돌합니다.

그래서 결정이 필요합니다.

1. Azure Speech를 예외적으로 외부 STT 서비스로 쓸 것인지
2. 아니면 Azure Speech 비용 문서는 참고만 하고, 온프레미스 STT 대안을 찾을 것인지

아직 안 된 것은:

- 실제 음성 인식 품질 검증
- 실제 통화 요약 품질 검증
- 실제 상담 답변 추천 품질 검증
- 실제 Azure Speech / Anthropic / OpenAI API key 기반 운영 테스트
- AI가 실패했을 때 어떻게 처리할지 정책
- AI 비용이 너무 많이 나가지 않도록 막는 정책

대표님께 보고할 때는 이렇게 말하는 것이 정확합니다.

> AI는 아직 서비스 기능으로 완성된 상태가 아닙니다.
> AI를 붙일 수 있는 기반은 만들었지만, 실제 Azure Speech STT와 LLM을 켜고 품질과 비용을 검증하는 작업이 남아 있습니다.
> 또한 Azure Speech는 외부 클라우드 서비스라서, 온프레미스 원칙과의 예외 여부를 결정해야 합니다.

## 5. 아직 안 된 것

### 5.1 실제 AI 기능

아직 운영 가능한 실제 AI 기능은 아닙니다.

남은 작업:

- Azure Speech STT 실제 연결 또는 온프레미스 STT 대안 확정
- Anthropic 또는 OpenAI 실제 연결
- 실제 통화 데이터로 요약 품질 확인
- 실제 통화 중 추천 품질 확인
- AI 실패 시 대체 동작 정리
- 비용 폭주 방지 장치 정리

### 5.2 실제 결제

현재는 요금제 제한과 사용량 화면은 있지만, 실제 결제는 없습니다.

아직 없는 것:

- 카드 결제
- 구독 결제
- 결제 실패 처리
- 영수증 / 인보이스
- Stripe 또는 Toss 연동

### 5.3 통화 녹음

통화 녹음 기능은 이제 시작했습니다.

현재 한 일:

- 녹음 파일 정보를 저장할 DB 구조를 만들기 시작함

아직 없는 것:

- 실제 녹음 파일 저장
- 온프레미스 저장소 연결
- 녹음 업로드
- 녹음 재생
- 녹음 다운로드
- 오래된 녹음 자동 삭제

### 5.4 실제 운영 이메일 테스트

이메일 발송 구조는 있지만, 운영 도메인 테스트가 필요합니다.

필요한 것:

- Resend 계정
- 발신 도메인 인증
- SPF / DKIM / DMARC 설정
- 실제 인증/초대/비밀번호 재설정 메일 수신 확인

### 5.5 일부 화면은 아직 데모 데이터

일부 화면은 아직 실제 API가 아니라 데모 데이터입니다.

예:

- Daily 화면
- Newsletter 화면
- 일부 dashboard 보조 섹션

## 6. 현재 막 시작한 작업: Phase 8 통화 녹음

Phase 8은 **통화 녹음 파일 저장 및 재생 기능**입니다.

현재 한 작업은 첫 단계입니다.

쉽게 말하면:

> 녹음 파일 자체를 저장한 것이 아니라,
> "어떤 통화에 어떤 녹음 파일이 연결되는지"를 기록할 DB 구조를 만든 상태입니다.

아직 실제 녹음 파일을 저장하거나 재생하는 기능은 없습니다.

온프레미스 원칙에 따라 녹음 저장소는 다음 중에서 결정해야 합니다.

- 온프레미스 MinIO
- 온프레미스 NAS
- 온프레미스 파일 서버

AWS S3 같은 public cloud 저장소는 사용하지 않는 전제로 정리합니다.

다음에 해야 할 일:

1. 온프레미스 녹음 파일 저장소 결정
2. 저장소 연결 코드 작성
3. 녹음 업로드 API 만들기
4. 녹음 재생 API 만들기
5. 화면에서 재생할 수 있게 만들기
6. 90일 지난 녹음 자동 삭제

## 7. 대표님 결정이 필요한 것

### 7.1 실제 AI를 먼저 살릴지 결정

현재 제품의 핵심 가치가 AI라면, 다음 우선순위는 AI가 되어야 합니다.

결정할 것:

- AI 기능을 지금 최우선으로 살릴지
- 아니면 통화 녹음이나 결제를 먼저 할지
- STT는 Azure Speech를 예외적으로 쓸지
- 아니면 온프레미스 STT 대안을 찾을지
- Anthropic / OpenAI 중 어떤 LLM을 먼저 쓸지
- API 비용 한도는 얼마로 둘지
- AI 품질 기준을 어떻게 볼지

### 7.2 결제 업체 결정

결제 연동을 하려면 업체를 정해야 합니다.

선택지:

- Stripe
- Toss Payments
- 둘 다

### 7.3 녹음 저장소 결정

통화 녹음을 저장하려면 파일 저장소가 필요합니다.

인프라 원칙:

- public cloud 사용 안 함
- 온프레미스 저장소 기준

선택지:

- 온프레미스 MinIO
- 온프레미스 NAS
- 온프레미스 파일 서버

### 7.4 통화 녹음 동의 정책

통화 녹음은 민감한 기능입니다.

결정할 것:

- 고객에게 녹음 안내를 할지
- 안내 문구를 어떻게 할지
- 녹음 보관 기간을 90일로 할지
- 누가 녹음 파일을 들을 수 있을지

### 7.5 운영 이메일 도메인

실제 서비스처럼 이메일을 보내려면 도메인 인증이 필요합니다.

필요한 것:

- 발신 도메인
- Resend 계정
- DNS 설정

## 8. 추천 우선순위

제 추천은 다음 순서입니다.

### 1순위: 실제 AI 기능 살리기

이 제품의 핵심이 AI라면, 먼저 실제 AI가 돌아가야 합니다.

해야 할 일:

- Azure Speech를 실제로 쓸지, 온프레미스 STT를 찾을지 결정
- 실제 STT 연결
- Anthropic 또는 OpenAI 실제 연결
- 실제 통화 데이터로 요약 품질 확인
- 실제 통화 중 추천 품질 확인
- AI 실패 시 대체 동작 정리
- 비용 폭주 방지 장치 정리

### 2순위: 통화 녹음

AI 검증에 실제 통화 데이터가 필요하다면 녹음 기능도 중요합니다.

해야 할 일:

- 온프레미스 저장소 연결
- 녹음 업로드
- 녹음 재생
- 녹음 보존/삭제 정책

### 3순위: 실제 결제

요금제 제한은 이미 들어갔기 때문에, 결제는 그 다음 단계로 붙이면 됩니다.

해야 할 일:

- Stripe 또는 Toss 결정
- checkout 만들기
- 결제 webhook 처리
- 구독 상태 동기화

## 11. 최종 정리

현재 Kloser는 **SaaS 관리 시스템의 기반은 꽤 많이 만들어졌지만, 대표 상품 가치인 실제 AI 기능은 아직 남아 있는 상태**입니다.

또한 앞으로의 인프라는 **public cloud가 아니라 온프레미스 기준**으로 잡아야 합니다.

따라서 대표님께 드릴 핵심 보고는 이것입니다.

> "계정, 고객, 팀, 통화 기록, 보고서, 보안, 감사 로그, 요금제 제한 같은 기본 SaaS 기반은 많이 완성됐습니다.
> 하지만 실제 AI 기능은 아직 완성된 상태가 아닙니다.
> STT는 Azure Speech 비용 가이드를 기준으로 검토하되, Azure Speech는 외부 클라우드 서비스라 온프레미스 원칙과 충돌합니다.
