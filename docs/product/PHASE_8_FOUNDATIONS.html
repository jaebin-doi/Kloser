PHASE 8 · 통화 녹취 · 저장 · 재생 · 자동 보존

통화 음성을 안전하게 —
저장하고, 재생하고, 90일 뒤 자동으로 정리까지

Phase 1~7로 통화 메타데이터·전사·요약·액션·보고서·운영 게이트는 모두 닫혔지만, 정작 "통화 음성 그 자체"를 다루는 자리가 비어 있었습니다. Phase 8은 그 자리를 한 묶음으로 채웁니다 — 녹취 metadata를 회사 단위로 분리해 저장하고, 음성 파일은 별도 저장소에만 보관하고, 통화 페이지에서 짧은 1회용 링크로만 재생하고, 90일이 지나면 자동으로 삭제합니다.

⌜ 핵심은 "음성 파일은 데이터베이스에 두지 않는다"입니다. DB는 metadata만, 음성은 외부 저장소만, 사용자는 짧은 만료의 1회용 링크로만 접근합니다. 다른 회사의 녹취는 어떤 경로로도 보거나 만질 수 없고, 운영 기록(감사 로그)에는 "몇 건이 어떻게 처리됐다"는 총계만 남아 누구의 어느 통화인지 추적이 불가능합니다. ⌟

한눈에 보기

  • 쉽게 말하면 — Phase 7까지가 "운영 출시 게이트"였다면, Phase 8은 "녹음한 통화를 안전하게 다루는 표면". 저장·재생·삭제·자동 보존이 한 묶음.
  • 왜 중요한가 — 통화 음성은 가장 민감한 자산입니다. 자격증명·평문 링크·파일 위치를 사용자나 로그에 노출시키지 않고, 다른 회사가 절대 접근할 수 없게 격리해야 유료로 팔 수 있습니다.
  • 사용자가 체감하는 변화 — 통화 상세 패널 안에 "녹음" 카드가 새로 보입니다. 재생 가능 / 처리 중 / 실패 / 없음 등 상태별 안내 + 1회용 재생 컨트롤 + (권한 있는 사용자) 삭제 버튼.
  • 자동으로 일어나는 일 — 업로드 후 90일이 지나면 음성 파일과 metadata가 자동 정리됩니다. 운영 기록에는 "총 N건 삭제" 같은 총계만 남고, 어느 통화 / 어느 파일인지는 절대 적히지 않습니다.
  • 아직 안 한 것 — 데스크탑/브라우저에서 직접 녹음 → 업로드 UI는 별도 트랙. 통화 동의(Consent) UX, 실 S3/MinIO 운영 smoke, 파형(waveform)·자막 시점 동기화는 v2 후보.

Phase 8에서 새로 가능해진 것

5가지 — 클릭하면 화면 모형과 함께 설명

1 · 녹취 metadata 분리

DB에는 metadata만, 음성 byte와 자격증명은 절대 저장 안 함. 회사 단위로 완전 격리.

2 · 저장소 + 1회용 링크

local / S3 / MinIO 3 모드. 사용자는 5~15분짜리 임시 링크로만 접근. 자격증명 화면 노출 0.

3 · 5종 REST 표면

업로드 · 확정 · 목록 · 재생 링크 · 삭제. 모든 동작이 권한 검사 + 감사 로그를 거침.

4 · 통화 페이지 재생

6가지 상태(없음/처리 중/재생 가능/실패/삭제됨/로딩) 카드 + 자동 링크 갱신.

5 · 90일 자동 보존

90일 지난 녹취는 음성 + metadata가 자동 정리. 감사 로그에는 총계만, 식별 정보 0.

+ 보너스 · 보안 위생 4가지

경로 가로채기 2단 방어 · 감사 총계화 · 자격증명 fail-fast · 화면 XSS 차단.

기능 01 · 데이터베이스 / RLS

녹취 metadata 분리 — 음성 byte는 절대 DB에 두지 않습니다

Phase 1~7까지의 어떤 테이블에도 통화 음성을 담을 자리가 없었습니다. Phase 8은 통화 녹취 metadata 전용 테이블을 새로 만듭니다. 이 테이블에는 "어느 회사의 어느 통화에 어떤 형식의 음성 파일이 어디 저장됐는지, 언제 업로드됐고 언제 삭제될 예정인지" 같은 정보만 들어가고, 음성 byte나 1회용 링크나 저장소 자격증명은 절대로 들어가지 않습니다.

테이블이 담는 것 / 담지 않는 것

✓ 담는다

  • · 회사 id · 통화 id · 녹취 id
  • · 상태 (업로드 대기 / 업로드됨 / 재생 가능 / 실패 / 삭제 대기 / 삭제됨)
  • · 음성 형식 (audio/webm 등 5종) · 길이 · 용량 · 코덱
  • · 업로드 시각 · 녹화 시각 · 보존 만료 시각
  • · 실패 사유(서버가 사전에 정리한 메시지만)

✗ 절대 담지 않는다

  • · 음성 byte (1바이트도)
  • · 1회용 재생/업로드 링크
  • · 저장소 자격증명 (access key 등)
  • · 저장소 endpoint / bucket 이름
  • · 다른 회사의 음성 정보 (RLS가 차단)

회사·통화 짝 외래키

녹취가 가리키는 통화는 같은 회사 안의 통화여야만 합니다. 다른 회사 통화 id를 끼우려고 하면 DB가 22503으로 거부합니다.

통화가 지워지면 함께 정리

통화가 사라지면 그 통화의 모든 녹취 metadata도 자동으로 사라집니다(ON DELETE CASCADE). 외부 음성 파일은 보존 정책 워커가 정리합니다.

조회용 인덱스

"이 통화의 녹취", "이 회사의 상태별 녹취", "보존 만료 시각으로 정렬한 녹취" 3가지 패턴에 맞춰 인덱스가 미리 만들어져 있습니다. 운영 데이터가 쌓여도 빠릅니다.

기능 02 · 외부 저장소 + 1회용 링크

저장소 + 1회용 링크 — 사용자에게 자격증명을 절대 주지 않습니다

통화 음성은 데이터베이스 바깥의 객체 저장소에 보관합니다. 사용자는 저장소 자격증명을 갖지 않고, 대신 서버가 5~15분짜리 1회용 링크를 그때그때 발급합니다. 만료 직전이 되면 자동으로 새 링크가 발급되어 재생이 끊기지 않습니다.

저장소 모드 3종

LOCAL

로컬 파일시스템 — 개발·테스트 기본값. 네트워크 호출 없음. 경로 가로채기 2단 방어로 시스템 다른 폴더에는 절대 쓰지 못함.

S3

AWS S3 — 운영 환경. 필요한 환경 변수(bucket·region·access key 등)가 누락되면 서버가 부팅 자체를 거부합니다(silent fallback 금지).

MINIO

MinIO (S3 호환) — 자체 운영 옵션. S3와 동일 인터페이스 + endpoint 추가. 자격증명 누락은 동일하게 부팅 거부.

1회용 재생 링크의 수명 — 만료 30초 전 자동 갱신

발급 (0초)
자동 갱신 (270초)
만료 (300초)

▲ 재생 링크 기본 수명은 5분(300초)입니다. 만료 30초 전에 화면이 자동으로 새 링크를 받아 교체하므로 재생이 끊기지 않습니다. 통화 상세를 닫거나 다른 통화로 넘어가면 그 즉시 갱신 타이머가 취소되고 음성 src가 비워집니다.

자격증명은 화면·로그·감사 어디에도 노출 0

access key / secret / bucket / endpoint는 서버 env에서만 읽고, 오류 메시지에 값이 그대로 찍히지 않습니다. 누락 시 메시지에는 "어떤 키가 빠졌는지" 이름만 나오고 값은 안 나옵니다.

경로 가로채기 2단 방어 (로컬 모드)

.. / encoded traversal / 절대경로 / 백슬래시 / 제어문자 / 1024자 초과를 모두 입력 단계에서 거부 + 실제 파일 시스템 쓰기 직전에 한 번 더 검증. 두 단계가 모두 통과해야만 디스크에 닿습니다.

업로드 파일 크기 250MB 상한

기본 250MB까지 허용합니다 (env로 조정 가능). 초과 요청은 413으로 사전 거부 — runaway upload가 저장소 비용을 폭주시키는 일이 없습니다.

기능 03 · 백엔드 API

5종 REST 표면 — 권한 + 감사 + 상태 검증을 한 번에

클라이언트가 저장소에 직접 닿지 않도록 모든 동작은 백엔드를 거칩니다. 5종 endpoint 각각이 인증 → 회사 컨텍스트 → 역할/소유권 확인 → 상태 검증 → 1회용 링크 발급 또는 metadata 변경 → 감사 로그를 한 묶음으로 처리합니다. 다른 회사의 통화 id를 보내면 단순히 404가 나오고, 어떤 정보도 누설되지 않습니다.

한 번의 업로드 흐름

1

업로드 시작 요청 — 클라이언트가 형식·용량을 알려 주면 서버가 metadata 행을 만들고 5~15분짜리 1회용 PUT 링크를 반환합니다. 이 링크는 그 한 파일에 대해서만 유효합니다.

2

음성을 저장소에 직접 PUT — 클라이언트가 받은 링크로 음성을 직접 올립니다. 서버를 거치지 않으므로 대용량 파일도 백엔드 메모리를 잡지 않습니다.

3

업로드 확정 — 올라간 byte의 실제 크기·체크섬을 서버에 알리면 검증 후 상태가 "재생 가능"으로 바뀝니다. 통화 페이지에서 즉시 재생 가능해집니다.

4

감사 로그 행 5종이 자동 적재 — 업로드 시작 · 확정 · 재생 링크 발급 · 삭제 요청 · 삭제 완료. 각 행에는 녹취 id·형식·길이·용량 정도만 들어가고 1회용 링크·파일 위치·자격증명은 절대 들어가지 않습니다.

다른 회사는 항상 404

cross-org 시도(다른 회사의 통화 id·녹취 id)는 권한 거부가 아니라 단순 404로 응답합니다. 존재 여부 자체를 노출하지 않습니다.

잘못된 상태 전이는 409

이미 재생 가능 상태에서 다시 finalize, 이미 삭제된 녹취 재삭제 등은 409로 거부합니다. 현재 상태를 함께 알려 주므로 클라이언트가 화면을 재정렬할 수 있습니다.

삭제는 2단계

먼저 metadata를 "삭제 대기"로 표시 → 저장소에서 파일 삭제 → metadata "삭제됨"으로 마무리. 중간에 실패해도 보존 정책 워커가 자동 재시도합니다.

기능 04 · 통화 페이지 상세 패널

통화 페이지 재생 — 상태별 6가지 화면

통화 페이지의 우측 상세 패널 안에 "녹음" 카드가 새로 생겼습니다. 같은 카드가 녹취 상태에 따라 6가지 모습으로 바뀝니다. 사용자는 어떤 상태든 "지금 무엇이 일어나고 있는지"를 한 줄로 알 수 있습니다.

▲ "재생 가능" 카드만 음성 컨트롤·URL 새로고침·삭제 버튼이 보입니다. 길이·용량·코덱 같은 메타는 서버가 보낸 값을 그대로 문자열로 박지 않고 한 단계 정리해서 표시합니다.

1회용 링크는 화면 텍스트로 보이지 않음

재생 가능 카드의 음성 컨트롤은 1회용 링크를 DOM 속성으로만 받습니다. 페이지 화면 텍스트, 콘솔, 페이지가 직접 쓴 HTML 어디에도 링크가 노출되지 않습니다.

다른 통화로 넘어가면 즉시 정리

상세 패널을 닫거나 다른 통화로 전환하면 음성 src·갱신 타이머·대기 중이던 응답이 즉시 무효화됩니다. 잘못된 통화의 링크가 다음 화면에 적용되는 일이 없습니다.

조회 권한은 삭제 버튼 안 보임

조회 전용 권한은 삭제 버튼이 표시되지 않습니다. 화면 가리기는 편의일 뿐이고, 권한 차단은 서버가 직접 합니다 — 직접 호출해도 403입니다.

기능 05 · 백그라운드 자동 정리

90일 자동 보존 — 약속이 코드로 강제됩니다

"통화 녹음은 90일 후 자동 삭제" 같은 보존 약속이 더 이상 문서뿐이 아니라 백그라운드 워커로 자동 실행됩니다. 운영자가 따로 작업하지 않아도 90일이 지난 녹취는 음성 파일과 metadata가 함께 정리됩니다. 감사 로그에는 "몇 건이 어떻게 처리됐다"는 총계만 남고, 어느 녹취·어느 통화인지 식별 정보는 절대 적히지 않습니다.

▲ 업로드 시각 + 90일이 기본 만료입니다. 특수한 경우(법적 보존 요구 등)에 한해 행마다 별도 만료 시각을 지정할 수도 있고, 두 조건 중 먼저 도래하는 시각에 자동 정리됩니다.

▲ 90일이 지난 녹취 3개를 정리했고 그중 1개는 저장소에 이미 파일이 없었다(idempotent 성공), 모두 S3 모드였다 — 정도만 남습니다. 어느 통화·녹취·파일 위치인지는 적히지 않습니다.

recording_id recording_ids call_id call_ids object_key storage_bucket signed_url playback_url upload_url checksum object_version access_key secret_access_key raw audio

▲ 자동화 테스트가 매 빌드마다 위 14개 토큰 중 하나라도 감사 로그 payload에 포함되면 빌드가 실패하도록 잠가 두었습니다. 실수로라도 누설될 수 없습니다.

저장소 호출은 DB 트랜잭션 밖

외부 저장소 호출은 DB 트랜잭션 안에서 하지 않습니다. 외부 응답이 느려도 DB 연결이 잡혀 있지 않으므로 다른 작업이 막히지 않습니다.

실패는 다음 사이클로 이월

저장소 응답이 일시적으로 실패한 행은 같은 사이클 안에서 재시도하지 않고 다음 사이클로 넘깁니다. 무한 재시도로 카운터가 부풀거나 호출이 낭비되는 일이 없습니다.

사용자 삭제 실패도 자동 재시도

사용자가 누른 삭제 버튼이 저장소 일시 오류로 끝나지 못해도 metadata는 "삭제 대기"로 남고, 다음 보존 사이클이 자동으로 재시도합니다. 사용자가 다시 누를 필요 없음.

기능 06 · 보안 위생

보너스 — 보안 위생 4가지

앞의 5가지가 외부 사용자나 운영자가 직접 느끼는 변화라면, 다음 4가지는 보안 사고를 처음부터 닫아 두는 위생 작업입니다.

경로 가로채기 2단 방어

로컬 모드에서 파일 위치 문자열에 .. / 인코딩된 traversal / 절대경로 / 백슬래시 / 제어문자 / 1024자 초과 같은 위험 패턴이 있으면 입력 단계에서 거부됩니다. 그래도 통과한 경우라도 실제 파일 쓰기 직전에 한 번 더 검증해, 두 단계가 모두 통과해야만 디스크에 닿습니다.

✔ 입력 정규식 거부 + 경로 정규화 후 재검증
✔ 다른 폴더 또는 OS 보호 영역에는 절대 쓰지 못함
✔ 에러 메시지에 위험한 입력값을 그대로 echo 안 함

감사 로그 총계화

보존 정책이 처리한 결과는 회사당 한 줄 총계만 남습니다. 어느 녹취·어느 통화·어느 파일이 정리됐는지는 적히지 않습니다. 감사 로그가 새는 경로 자체를 닫았습니다.

✔ 식별 가능한 값(녹취 id·통화 id·파일 위치)은 payload 0건
✔ 자동화 테스트가 14개 위험 토큰 substring을 매번 검사
✔ 한 사이클에 0건 처리된 경우 행을 만들지 않아 노이즈 차단

자격증명 fail-fast

운영 환경에서 S3/MinIO 모드를 지정했는데 필수 환경 변수(bucket·region·access key 등)가 누락되면 서버가 부팅 자체를 거부합니다. 누가 어떤 키를 빠뜨렸는지 메시지에 이름만 적고 값은 절대 echo하지 않습니다.

✔ silent fallback 금지 — 부팅이 끝났다 = 모든 자격증명이 OK
✔ 메시지에 키 "이름"만, "값"은 0건
✔ dev 기본은 자동으로 local 모드 (네트워크 호출 0)

통화 페이지 XSS 차단

서버가 보낸 값(실패 사유 메시지, 코덱 이름, 길이·용량)은 모두 화면에서 텍스트로만 렌더되고, 페이지가 직접 쓴 HTML 템플릿에는 끼우지 않습니다. 1회용 재생 링크도 화면 텍스트로는 절대 노출되지 않습니다.

✔ 실패 사유 메시지: 서버가 사전 정리 + 화면에서 escape
✔ 1회용 링크: 음성 컨트롤 DOM 속성으로만 결합
✔ 길이·용량·코덱: 표시 단계에서 정규화 후 escape
