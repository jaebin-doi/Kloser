PHASE 7 · 운영 게이트 · 보안 · 감사 · 플랜

외부 사용자에게 유료로 팔기 직전에 필요한 —
이메일·보안·감사·보존·플랜 5가지를 한 묶음으로

Phase 6까지 통화·고객·보고서 핵심 기능은 닫혔지만, "실제로 메일이 도착하는가", "관리자 계정이 비밀번호 하나로만 보호되는가", "누가 무엇을 언제 했는지 추적할 수 있는가", "보관 정책이 코드로 강제되는가", "유료 플랜의 한도가 화면에 보이는가" 다섯 자리가 비어 있었습니다. Phase 7은 그 다섯 자리에 운영 출시 직전 게이트를 한꺼번에 채웁니다.

⌜ Phase 6이 "사용자 자동화"였다면, Phase 7은 "운영 자동화"입니다. 회원가입 메일이 진짜로 도착하고, 관리자 계정에 1회용 코드가 한 단계 더 붙고, 모든 운영 행위가 admin 패널에 기록되고, 3년 된 통화 기록이 자동으로 삭제되고, 플랜 한도를 넘으면 서버가 직접 막아 줍니다. ⌟

한눈에 보기

  • 쉽게 말하면 — 외부 사용자에게 유료로 팔기 직전에 필요한 운영 게이트 5종(실 이메일·2단계 인증·감사 로그·보존 정책·플랜 한도)을 한 묶음으로 닫은 단계입니다.
  • 왜 중요한가 — 운영 출시는 "기능"이 끝났다고 되는 게 아닙니다. 관리자 계정 탈취 차단, 운영 행위 추적, 개인정보 보존 정책 강제, 플랜 한도 강제 — 이게 안 되면 유료 도입이 불가능합니다.
  • 사용자가 체감하는 변화 — 로그인 시 6자리 코드 / 설정 → 보안에 감사 로그 패널 / 설정 → 플랜 & 사용량에 6 한도 진행률 + 청구 정보 form.
  • 남은 운영 검증 — 운영 도메인으로 실제 이메일 발송 smoke (Resend), 보존 정책 워커 staging 활성화 확인.
  • 아직 안 한 것 — 실 결제 provider(Stripe/Toss — 별도 Phase, 외부 돈 이동 동반), 통화 녹음(Phase 8 진행 중).

Phase 7에서 새로 가능해진 것

5가지 — 클릭하면 화면 모형과 함께 설명

1 · 실 이메일 발송

회원가입 인증·비밀번호 재설정·동료 초대가 실제 메일 서비스로 발송됩니다.

2 · 2단계 인증 (MFA)

TOTP 6자리 코드를 한 번 더 입력해야 로그인이 끝납니다. 조직 단위로 강제화 가능.

3 · 감사 로그

로그인·권한 변경·고객/통화/지식/청구 정보 변경이 admin 전용 패널에 한 줄씩 남습니다.

4 · 보고서 기간 + 담당자

7/30/90일 또는 직접 기간 선택, 담당자별 KPI + 행 클릭으로 통화 drill-down.

5 · 플랜 & 사용량

현재 플랜의 6가지 한도 진행률과 청구 정보를 admin 화면에서 직접 확인·수정.

+ 보너스 · 운영 위생 4가지

보존 정책 자동 실행 · 역할별 사이드바 정돈 · LLM 비용 환산 · demo 정직성 정리.

기능 01 · 회원가입 / 초대 / 비밀번호 재설정

실 이메일 발송 — 회원가입 메일이 진짜로 도착합니다

Phase 3까지 회원가입 인증·비밀번호 재설정·동료 초대 이메일은 개발용 임시 저장소에만 행으로 남았고, 평가용 e2e가 그 행에서 직접 토큰을 꺼냈습니다. Phase 7부터는 운영 환경에서 Resend 메일 서비스를 통해 실제 메일이 수신자 받은편지함에 도착합니다. 로컬 dev와 e2e는 기존 임시 저장소 모드를 그대로 유지하므로 회귀 위험이 없습니다.

메일이 도착하는 흐름

1

회원가입/초대/비밀번호 재설정 요청 시 발송 대기열에 기록됩니다. 이때 보안 토큰(평문 링크)은 별도 자리에 암호화 보관되고, 로그 archive에는 마스킹된 형태로만 남습니다.

2

발송 워커가 대기열에서 한 건씩 꺼내 Resend에 전달합니다. 요청 트랜잭션 안에서 외부 HTTP를 직접 호출하지 않기 때문에 응답이 느려지지 않습니다.

3

성공하면 평문 토큰을 폐기합니다. 일시 실패는 지수 백오프로 재시도하고, 끝까지 실패한 건은 dead letter 표시 후 평문 토큰을 같이 비웁니다.

로컬 dev / e2e 모드는 그대로

기본값은 임시 저장소 모드입니다. 로컬에서는 메일이 실제로 발송되지 않고 dev DB에 행으로 남아서 e2e 스크립트가 토큰을 꺼낼 수 있습니다.

평문 토큰은 archive에 남지 않음

발송 본문의 평문 링크는 암호화 보관 후 발송 성공/실패와 함께 폐기됩니다. 로그·archive에 평문 토큰이 검색되는 일이 없습니다.

기능 02 · 로그인 / 설정 → 보안

2단계 인증 — 관리자 계정 탈취 자체를 차단

관리자 비밀번호 하나로만 보호되는 조직은 비밀번호 유출 즉시 모든 데이터가 노출됩니다. Phase 7부터는 사용자가 Google Authenticator 같은 인증기 앱과 자기 계정을 묶어 두면, 다음 로그인부터 비밀번호 입력 직후 6자리 1회용 코드를 한 번 더 입력해야 로그인이 끝납니다. 관리자는 조직 단위로 모든 멤버에게 강제화하는 토글도 가집니다.

등록 화면 모형 — QR 스캔 후 6자리 코드 확인

인증기 앱으로 QR을 스캔하거나, 아래 보조 키를 수동 입력합니다.

JBSWY3DPEHPK3PXP

인증기가 보여 준 코드

4 8 1 5 6 2

화면이 보여 주는 6자리를 그대로 입력하면 등록이 끝납니다. 이후 모든 로그인은 비밀번호 + 이 6자리 코드 두 단계입니다.

조직 강제화 토글

관리자는 설정 → 보안에서 "조직 전체 2단계 인증 필수화"를 켤 수 있습니다. 켜진 순간부터 미등록 멤버는 다음 로그인 시 등록이 강제됩니다.

5분짜리 일회용 토큰

등록·해제 직전에 현재 비밀번호를 한 번 더 확인합니다. 그 확인은 5분 안에만 유효한 일회용 토큰으로 묶여 있어, 누가 옆에서 폼만 가로채도 다시 비밀번호를 알아야 합니다.

실패 누적 시 자동 잠금

6자리 코드를 연속으로 잘못 입력하면 잠시 잠깁니다. 잠금이 풀리는 데 시간이 걸리므로 무차별 대입 시도를 효과적으로 차단합니다.

기능 03 · 설정 → 보안 → 감사 로그

감사 로그 — 누가 언제 무엇을 했는지 한 줄씩

Phase 6까지는 권한 변경·삭제 같은 운영 행위가 일어나도 그 자취를 추적할 수 있는 공식 화면이 없었습니다. Phase 7부터는 관리자가 설정 → 보안 하단의 감사 로그 패널에서 최근 운영 행위를 시간 역순으로 확인합니다. 값(고객 이름, 이메일, 청구 정보 등)은 절대 기록되지 않고 — "무엇이 바뀌었는가"의 필드 이름만 남습니다.

값은 기록 안 함, 필드 이름만

고객 이름, 이메일, 사업자등록번호, 비밀번호 — 값은 절대 audit row에 남지 않습니다. "어떤 필드가 바뀌었다"는 사실만 기록합니다.

조직 격리 완전 보장

다른 조직의 행위는 여기 절대 보이지 않습니다. 로그 자체가 조직 단위로 분리 저장되고, 화면도 본인 조직 행만 비춥니다.

XSS 안전

audit row의 모든 값은 화면에서 텍스트로만 렌더됩니다. 행위자 이름·필드 이름에 특수문자가 있어도 화면이 깨지거나 코드가 실행되는 일이 없습니다.

기능 04 · 팀 보고서 화면

보고서 정밀화 — 기간 선택 + 담당자별 drill-down

Phase 6의 팀 보고서는 "전체 기간"의 KPI만 보여 줬습니다. Phase 7부터는 최근 7일/30일/90일/직접 기간 중 하나를 골라 그 기간의 KPI를 봅니다. 담당자별 KPI가 새로 추가되어, 행을 클릭하면 그 담당자의 최근 통화만 골라서 빠르게 확인할 수 있습니다.

▲ 클릭된 행("김민수")이 강조되면, 아래 최근 통화 10건이 그 담당자의 통화로 필터됩니다. 미배정 버킷은 조직 범위(admin)에서만 보입니다.

매니저는 본인 팀, admin은 전체 조직

매니저는 본인 팀의 active 멤버만 담당자 목록에 보입니다. admin은 전체 조직 + 미배정 버킷까지 봅니다. 어느 쪽도 다른 팀의 통화를 볼 수 없습니다.

기간 default = 최근 30일

아무 preset도 고르지 않으면 자동으로 최근 30일을 봅니다. "전체 기간"이 기본이던 Phase 6과 달라진 점이라 의도된 변경입니다.

기능 05 · 설정 → 플랜 & 사용량

플랜 & 사용량 — 한도가 차면 서버가 직접 막아 줍니다

그동안 "플랜"은 가입할 때 정해진 라벨뿐, 한도가 코드로 강제되지 않았습니다. Phase 7부터는 admin이 설정 → 플랜 & 사용량에서 6가지 한도(좌석·고객사·지식베이스·지식 청크·이번 달 통화·이번 달 AI 비용)의 진행률을 한 화면으로 봅니다. 한도가 차면 다음 mutation 호출이 서버에서 차단됩니다 — 결제 약관과 화면이 일치합니다.

청구 정보

결제 연동은 아직 준비 중입니다. 플랜 변경이나 결제 처리는 운영팀(support@kloser.com)으로 문의해 주세요.

한도 차면 서버가 차단

좌석·고객·지식·통화 5종은 한도를 넘으면 다음 등록·생성 요청이 서버에서 거부됩니다. 화면에는 "플랜 한도 초과" 안내가 나타납니다.

AI 비용은 참고용

이번 달 AI 비용은 표시만 합니다. 비용이 한도를 넘어도 호출은 막지 않습니다 — admin이 사용 추세를 보고 플랜을 올릴 시점을 판단하기 위한 신호입니다.

청구 정보는 안전하게

사업자등록번호·세금계산서 이메일은 admin만 보고 수정합니다. 외부 결제 시스템 ID는 응답에 노출되지 않고, 감사 로그에는 "어떤 필드가 바뀌었는지"만 남아 값이 새지 않습니다.

기능 06 · 백그라운드 / 사이드바 / 운영자 콘솔

보너스 — 운영 위생 4가지

앞의 5가지가 외부 사용자에게 직접 보이는 변화라면, 다음 4가지는 운영자만 직접 체감하거나 백그라운드에서 조용히 정리되는 위생 작업입니다.

보존 정책 자동 실행

"통화 내용 3년 보관 후 삭제" 같은 보존 약속이 문서뿐만 아니라 백그라운드 작업으로 자동 실행됩니다. 발송 직전에 멈춘 메일도 일정 시간 이상 막혀 있으면 자동으로 재시도 가능 상태로 되돌아갑니다.

✔ 3년 지난 통화 내용: 자동 삭제
✔ 발송 멈춘 메일: 일정 시간 후 자동 회복
✔ 사이클당 1행 요약 기록 (몇 건 처리됐는지)

역할별 사이드바 정돈

상담원·조회 권한 사용자에게 admin 전용 메뉴(팀 보고서 등)가 사이드바에서 깔끔히 숨겨집니다. URL 직접 접근은 backend가 별도로 막습니다.
