# 1. RLS 사전에 적용하면 문제 해결?

**완전 해결은 아니고, “조직 데이터 유출”의 큰 사고를 막는 핵심 방어막이 생김.**

해야 할 구조:

```text
모든 주요 테이블에 org_id 추가
→ PostgreSQL RLS 활성화
→ 요청마다 DB에 current_org_id 설정
→ DB가 org_id 다른 row 자동 차단
```

즉:

```sql
SELECT * FROM customers;
```

실수로 이렇게 써도 DB가 내부적으로:

```sql
WHERE org_id = 현재_조직
```

처럼 막아줌.

결론:

```text
RLS + repository org_id filter = 이중 방어
```

문서에도 조직별 데이터 분리를 `org_id`로 하겠다고 되어 있으니, 이걸 코드만 믿지 말고 DB 정책까지 거는 게 맞음.

# 2. 서버는 어떻게 늘려?

단계별로 가면 됨.

## 지금 구조

```text
서버 1대
- app
- worker
- postgres
- redis
- nginx
```

문제:

```text
이 서버 죽으면 전체 다운
```

---

## 1단계: App 서버만 2대로 늘리기

```text
Load Balancer
   ├── App Server 1
   └── App Server 2
        ↓
     PostgreSQL 1대
     Redis 1대
```

효과:

```text
앱 하나 죽어도 다른 앱이 받음
```

---

## 2단계: DB 복제 추가

```text
PostgreSQL Primary
        ↓ 복제
PostgreSQL Replica
```

효과:

```text
Primary 죽으면 Replica를 승격해서 복구
```

---

## 3단계: 진짜 99.9%용 구조

```text
Load Balancer 2대
App Server 2대 이상
Worker 2대 이상
PostgreSQL Primary + Replica
Redis Sentinel or Redis HA
Off-site Backup
Monitoring
```

결론:

```text
단일 서버로는 SLA 99.9% 약속하지 말고,
Enterprise 전에 HA 전환 Phase를 넣어야 함
```

문서의 현재 배포안은 초기 단일 서버/systemd 또는 Docker Compose라서 운영 SLA와 충돌 가능성이 큼.

# 3. 검증된 Auth 라이브러리 사용이 뭐냐?

뜻은 이거야.

```text
로그인/회원가입/세션/토큰/비밀번호 재설정 같은 위험한 부분을
우리가 처음부터 직접 만들지 말고,
이미 검증된 라이브러리 위에서 구현하자
```

추천은 지금 기준으로 **Better Auth 우선 검토**가 맞아 보임. Better Auth는 TypeScript용 인증/권한 프레임워크이고, 2FA, passkey, multi-session, SSO 같은 기능을 플러그인 방식으로 제공한다고 공식 문서가 설명함. PostgreSQL 어댑터와 schema migration/generation도 지원함. ([Better Auth][1])

Lucia는 주의해야 함. 현재 Lucia 공식/깃허브 쪽은 “라이브러리”라기보다 JavaScript/TypeScript auth 구현 리소스 성격으로 바뀐 상태이고, v3는 deprecated 공지가 있음. 그래서 새 프로젝트라면 Better Auth 쪽이 더 현실적임. ([Lucia][2])

정리:

```text
비추천:
직접 sessions, refresh rotation, password reset 전부 구현

추천:
Better Auth + PostgreSQL
+ 우리 서비스의 organization/membership/role 로직만 얹기
```

# 4. 최소 운영 체계는 뭘 말해?

“서버 올렸다”가 운영이 아님. 최소한 이 5개가 있어야 함.

## 필수 5종 세트

```text
1. 백업
2. 복구 테스트
3. 모니터링
4. 로그/에러 추적
5. 배포 자동화
```

구체적으로:

```text
백업:
매일 DB backup
같은 서버 말고 NAS/S3/다른 서버에 저장
암호화
월 1회 restore drill

모니터링:
CPU/RAM/Disk/PostgreSQL/Redis/App 상태 확인

알림:
디스크 80% 이상
API 5xx 증가
DB 연결 실패
Worker job 실패
SSL 인증서 만료 전 알림

로그:
API access log
error log
audit log
worker log

배포:
git push 후 자동 build/test/deploy
migration before deploy
rollback 방법
```

문서에는 daily backup, health check, disk usage alert 등이 언급되지만 아직 도구와 절차가 구체화되어 있지 않음.

# 5. 조직별 사용량 제한은 어떻게 해결?

조직마다 “월 사용 한도”를 DB에 둬야 함.

예:

```text
무료 조직:
월 STT 300분
월 AI 100만 token

Pro 조직:
월 STT 3,000분
월 AI 1,000만 token

Enterprise:
계약별 설정
```

DB 테이블 예시:

```text
usage_quotas
- org_id
- month
- stt_minutes_limit
- ai_tokens_limit
- storage_gb_limit

usage_counters
- org_id
- month
- stt_minutes_used
- ai_tokens_used
- storage_gb_used
```

실행 흐름:

```text
1. STT/AI 요청 들어옴
2. 현재 조직 사용량 조회
3. 한도 초과면 요청 차단
4. 가능하면 실행
5. 사용량 증가 기록
```

결론:

```text
AI/STT는 무조건 org별 quota가 있어야 함
안 그러면 고객이 많이 쓸수록 회사가 손해 볼 수 있음
```

# 최종 정리

```text
1. RLS:
Phase 1부터 켜서 조직 데이터 유출 방어

2. 서버:
단일 서버 → App 2대 → DB Replica → HA 구조

3. Auth:
직접 구현 말고 Better Auth 우선 검토

4. 운영:
백업/복구/모니터링/로그/배포를 Phase 1에 넣기

5. 사용량 제한:
org별 STT분/AI token/storage quota 테이블 만들기
```
